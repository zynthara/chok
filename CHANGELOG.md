## [Unreleased]

### Added

- **store: aggregates â€” `Sum` / `Avg` / `Min` / `Max` / `CountDistinct` /
  `GroupBy`.** The front door for the dashboard reads that previously
  required `Unsafe`, as free functions beside the `Pluck` family and
  under exactly `Count`'s read semantics: fields resolve through the
  query allowlist (an undeclared field stays a raw
  `where.ErrUnknownField` â€” server code wrote the name, so it is a 500,
  not a client 400), scopes apply fail-closed, soft-deleted rows are
  excluded, and filter options narrow the aggregated set while
  pagination/ordering options are stripped exactly like `Count` strips
  them. Column kinds are schema-derived by the same wire-type probe the
  cursor encoder uses: `Sum`/`Avg` accept numeric columns only,
  `Min`/`Max` additionally accept time columns; string/bool columns are
  not aggregatable (MIN over text is collation-defined per dialect), and
  `CountDistinct` demands a comparable scalar (a PostgreSQL `json`
  column is refused up front â€” including a Go string field declared
  `gorm:"type:json"` â€” instead of failing mid-query; string cardinality
  follows the column's collation). On SQLite â€” which stores timestamps
  as text in the writer's zone and compares them lexicographically â€”
  aggregates read time columns through a UTC-normalising expression, so
  MIN/MAX pick the chronological (not lexicographic) extreme across
  mixed zone offsets and one instant written under two offsets groups
  and counts as one value, at millisecond precision (matching MySQL's
  DATETIME(3) write precision) and returned in UTC; numeric legacy
  storage reads as Unix seconds via an explicit typeof branch (never
  SQLite's 'auto' heuristic, which misreads early-1970 Unix seconds as
  Julian days). PostgreSQL timestamptz compares instants natively.
  MySQL time columns are DATETIME â€” wall clocks in the writing
  process's zone with no UTC conversion â€” so instants compare correctly
  only under chok's documented deployment invariant: every writer of
  one database runs in the same FIXED, transition-free zone (TZ=UTC
  recommended â€” a DST zone folds distinct instants into one wall clock
  at every fall-back, even within a single process); a fold or mixed-TZ
  write history cannot be repaired at read time, and the same invariant
  already bounds ordering, range filters and cursors. JSON columns are
  refused by the migrator-resolved dialect type as well, so custom
  types mapping to JSON via `GormDBDataType` cannot slip past the
  logical-type check.
  Results converge across the three dialects to caller-declared Go
  types: `Sum[int64]` is exact over integer columns and errors loudly
  past the int64 range (never silent truncation), `Sum[float64]` widens
  with the documented 2^53 trade, `Avg` is always `float64`, and SQL
  NULL (zero rows, or all inputs NULL) surfaces as `ok=false` â€” never
  conflated with a real zero. `GroupBy[K]` buckets by one allowlisted
  column (`K` must match its wire kind exactly; a NULL group key errors
  loudly and points at `where.WithFilterNotNull` instead of folding into
  the zero-value bucket) and computes positional `AggValue` results
  (`CountRows` / `CountDistinctOf` / `SumOf` / `AvgOf` / `MinOf` /
  `MaxOf`) with kind-checked accessors; groups come back ordered by key
  ascending, and non-filter options â€” `WithCount` included â€” are
  rejected: a silently dropped `WithOrder`+`WithLimit` would masquerade
  as top-N. Bool group keys accept only canonical 0/1 storage (a raw 2
  forms its own SQL bucket; folding it onto `true` would hand back
  duplicate keys), and a table-qualified column alias resolves when the
  qualifier names the model's own table. Ordering by
  aggregate values, `LIMIT`, `HAVING`, multi-column grouping and
  string `MIN`/`MAX` are deliberately absent from v1 (see
  `docs/db.md` Â§5.7); none of the new API touches `Reader` or any
  interface.

## [2.0.0-beta.6](https://github.com/zynthara/chok/compare/v2.0.0-beta.5...v2.0.0-beta.6) (2026-07-17)

### Added

- **store: declarative constraintâ†’field mapping for duplicate errors.**
  `store.WithConstraintFields(map[string]string)` declares which public
  field a unique constraint should blame (Ecto's `unique_constraint`
  idea). A declared hit populates the new `DuplicateEntryError.Field`,
  and `store.MapError` then reports response metadata `field` instead
  of the raw constraint name â€” index names leak schema layout and drift
  with migrations, field names are the API's own vocabulary. Undeclared
  constraints keep the existing `constraint` metadata unchanged. Keys
  match the driver-reported identifier with table qualifiers stripped
  (MySQL 8 reports `table.key`); SQLite names no index â€” it reports the
  violated column list (`delete_token` included for `SoftUnique`
  indexes) â€” so cross-dialect declarations list both spellings.

- **store: `ListIn` â€” the auto-chunked second half of the two-step IN
  pattern.** `store.ListIn(ctx, s, field, values, opts...)` retrieves
  every row whose field value is in the set, deduplicating the values
  and splitting them into IN lists of at most `where.MaxInList`
  automatically â€” the blessed cross-table pattern no longer hands the
  chunk loop to the caller. The anchor is one big IN's SET semantics:
  when the set spans multiple chunks, combined rows are additionally
  deduplicated by primary key, because database equality can be wider
  than Go equality (a case-insensitive collation matches one row from
  values in two different chunks). Each chunk runs under exactly
  List's read path (query allowlist, scopes, soft-delete rules), and
  an empty value set still runs one degenerate no-match query so the
  allowlist, the option guard and fail-closed scopes all apply â€”
  never a silently successful empty page. Options are filters-only
  (ordering, pagination, count and page-size caps do not compose
  across chunks and are rejected), the result carries no order
  guarantee, and â€” like `ListByIDs` â€” it is server-side plumbing sized
  by the value set, deliberately not subject to the Store's
  max-page-size cap (a per-chunk clamp would silently drop rows
  mid-set). Multiple chunks are multiple SELECTs, not one statement,
  and a transaction restores single-snapshot semantics only where the
  default isolation provides a transaction-wide snapshot (`Store.Tx` /
  `db.RunInTx` use the database default): SQLite and MySQL/InnoDB
  qualify; PostgreSQL's default READ COMMITTED re-snapshots per
  statement â€” escalate to REPEATABLE READ yourself (via the Unsafe
  door, first statement in the transaction) when cross-chunk snapshot
  consistency matters there.

- **store: pessimistic locking via `GetForUpdate`.** The transactional
  counterpart to Update's optimistic locking:
  `GetForUpdate(ctx, locator, opts...)` reads a row under
  `SELECT ... FOR UPDATE`, holding off concurrent lockers and writers of
  that row until the enclosing transaction commits. The entry point is
  verifiable by construction: it requires a transaction on the store's
  own handle (`ErrLockRequiresTx` otherwise â€” a lock under autocommit is
  released before the caller can use it), refuses read-only stores
  (`db.ErrReadOnly`) and rejects `WithPreload` (`ErrLockPreload` â€”
  association queries run outside the lock). On SQLite the driver drops
  the locking clause, but chok's SQLite shape serializes every
  transaction and write onto a single write connection (file databases
  pin a one-connection write pool, `_txlock=immediate` by default;
  memory databases run on one pinned connection), so the enclosing
  transaction excludes every concurrent writer â€” stronger than a row
  lock â€” and the observable guarantee is uniform across dialects.

- **where: `WithCursorByField`.** The composite-cursor option whose
  tie-breaker is a caller-chosen column: the sort field resolves through
  the query allowlist (client-facing input), while the tie column is
  identifier-validated but deliberately not allowlist-resolved â€” it is
  trusted infrastructure, which is how `ListWithCursor` binds the model's
  RID column without requiring "id" in the allowlist. `WithCursorBy`
  (internal id tie-breaker) remains the server-side DSL tool.

- **store: unified admin-role policy.** `db.store.admin_roles` (and
  `db.instances.<name>.store.admin_roles`) plus the per-store
  `store.WithAdminRoles(...)` option name the principal roles that bypass the
  automatic OwnerScope AND may set OwnerID explicitly on create. One list â€”
  resolved as option > policy > deprecated package default and captured at
  store construction â€” now drives both sides, closing two authorization gaps:
  the write-side owner fill used to re-read the mutable global list per call
  (so `SetDefaultAdminRoles` after construction skewed write semantics away
  from the already-frozen query scope), and the previously documented
  per-store override (`WithScope(OwnerScope("x"))`) actually intersected
  bypass sets instead of replacing them because scopes compose by AND.
  `WithAdminRoles()` with no arguments removes every bypass (fail-closed);
  `SetDefaultAdminRoles` remains as the deprecated fallback default.

- **db: append-only migration repair history.** Every repair â€” application
  and owned-sequence `retry` / `mark-applied` / `accept-drift`, and manifest
  claim transfers â€” now persists its full evidence (action, checksums or
  owners, mandatory reason, operator, chok version) to
  `schema_migrations_chok_repairs` in the repair's own transaction: a history
  row means the business-state compare-and-swap committed, and a repair whose
  history cannot be recorded fails instead of committing. `db.RepairHistory`
  reads the evidence most-recent-first with kind filtering and clamped
  limits; `app` and `repairs` join `manifest` as permanently forbidden
  sequence kinds, and `OwnedSequence` now resolves kind legality through the
  same `ValidateSequenceKind` gate as every other consumer.

- **db: third-party owned-sequence manifest and claim protocol.**
  `db.ManifestEntries`, `db.LedgerSnapshot` and `db.RepairSequenceClaim` expose
  a read-only catalog, file-independent ledger health and lock-protected owner
  transfer, and `db.ValidateSequenceKind` is the shared gate tooling uses
  before deriving ledger identifiers from externally observed kind strings. Every owned sequence persists its component-package owner,
  compatibility engine floor and informational component/chok versions in
  `schema_migrations_chok_manifest`. Apply, repair and claim transfer authorize
  under the migration lock; legacy ledgers are adopted only after validation
  and before any subsequent ledger or schema write. `chok migrate status`
  discovers both claimed and legacy-unclaimed ledgers, while strict `--check`
  fails on unclaimed or file-unverified third-party content unless
  `--ledger-health-only` is explicitly selected.

- **audit/authz: exported auto-schema helpers.** `audit.MigrateSchema` and
  `authz.MigrateSchema` expose each battery's sanctioned AutoMigrate path â€”
  the exact calls their modules make under `migrate: auto` â€” mirroring the
  existing `account.MigrateSchema`. Kernel-less embedders and the migration
  behavior fixtures now share one schema source instead of copying raw
  AutoMigrate calls that could drift from the module.

- **store: per-row batch update and atomic batch upsert.** `BatchUpdate`
  applies `Fields` semantics, optimistic locking and scopes to per-object
  payloads in one transaction, restoring framework-mutated versions when the
  batch fails. `BatchUpsert` validates conflict/update columns and duplicate
  conflict tuples before SQL, then executes all GORM chunks atomically. The
  opt-in `BatchWriter[T]` interface exposes both without expanding the existing
  narrow `Writer[T]` contract.

### Changed

- **store: unknown-field errors are classified by entry point.**
  **Breaking:** the programmatic read entry points â€” `List`, `ListQ`,
  `Count`, `Pluck`/`PluckDistinct`/`PluckIDs`, `ListIn`,
  `ListWithCursor`'s cursor field, and the locator-based `Get` /
  `GetForUpdate` / `Update` / `Delete` / `Restore` / `Exists` â€” now
  return `where.ErrUnknownField` raw (chain intact, `errors.Is` works;
  `store.MapError` does not map it â†’ 500) instead of pre-mapping it to
  `apierr.ErrInvalidArgument` (400). Field names on these entry points
  are written by server code, so `List(ctx, where.WithFilter("typo",
  v))` is a programming bug that must alarm monitoring â€” the old
  mapping disguised it as a client mistake. The `ListFromQuery` chain
  keeps the 400 mapping end to end: there the field names arrive in
  the URL and genuinely are client input. VALUE errors
  (`where.ErrInvalidParam` â€” pages, sizes, cursor tokens, filter
  values) keep their 400 mapping on every entry point, so handlers
  may still pass client pagination values straight into `WithPage` /
  `WithLimit` / cursor size. Migration: route client-supplied FIELD
  names through `ListFromQuery` (or validate them against the
  allowlist in the handler) instead of splicing them into
  `WithFilter` / `WithOrder`; code that matched the old 400 via
  `errors.Is(err, apierr.ErrInvalidArgument)` on programmatic calls
  should match `errors.Is(err, where.ErrUnknownField)`. The update
  side is unchanged (`ErrUnknownUpdateField` was already a 500).

- **store: cursor size discipline.** Two documented bounds join the
  opaque-cursor contract as exported constants: `MaxCursorTokenLen`
  (4096 bytes) caps the token `ListWithCursor` accepts â€” over-limit
  tokens are rejected as `apierr.ErrInvalidArgument` before any
  base64/JSON work, so token length can no longer buy proportional
  server-side decode allocation â€” and `MaxCursorValueLen` (1024 bytes)
  caps the string representation of a boundary value the encoder will
  sign. A page whose proven-to-exist next boundary exceeds the value
  bound (or whose assembled token would exceed the token bound â€” JSON
  escaping can inflate control-character-heavy strings past it) returns
  a server-side error instead of signing a token the framework itself
  refuses to decode, and never silently truncates the value. Previously
  both sides were unbounded. Cursor fields are short scalar keys; if a
  column can legitimately exceed 1KB, it should not key a cursor.

- **store/handler: `ListFromQuery` returns `*Page[T]` and leaves the
  `Reader` contract.** The list envelope now lives in the query layer as
  `where.Page[T]` â€” `store.Page` stays as an alias â€” so
  `handler.QueryLister` speaks the same envelope without handler
  importing store. **Breaking:** `Store.ListFromQuery` (and
  `handler.QueryLister`) returns `(*Page[T], error)` instead of
  `([]T, int64, where.PageInfo, error)`; callers read `page.Items` /
  `page.Total` / `page.Meta`. **Breaking:** `store.Reader` no longer
  includes `ListFromQuery` â€” the read contract is transport-free;
  parsing URL input belongs to the edges (`handler.HandleList`, or the
  `*Store.ListFromQuery` sugar, which both keep working). Nothing
  changes at the HTTP surface.

- **store: `Writer` is the single-row write surface.** The
  Writer/BatchWriter split now follows a single line â€” single-row
  versus multi-row â€” instead of its historical accretion. `Upsert`
  (single row) stays on `Writer`. **Breaking:** `BatchCreate` leaves
  `Writer` (and therefore `ReadWriter`); it remains on `BatchWriter`
  and on `*Store`. Code that batch-creates through a `Writer[T]`
  dependency declares `BatchWriter[T]` (or keeps `*Store[T]`).

### Fixed

- **db: table-existence probes no longer fabricate verdicts or results
  out of failed queries.** gorm's `Migrator().HasTable` swallows query
  errors into `false`, so a probe running under an expired or cancelled
  context read as "table absent" â€” which the migration surfaces
  escalated into fabricated verdicts (`RepairSequenceClaim` reported
  `ErrSequenceManifestCorrupt` "claim exists without ledger" /
  `ErrSequenceUnclaimed`; `SchemaFingerprint` reported a missing table)
  or result-shaped lies (`ManifestEntries` returned an empty manifest,
  `LedgerSnapshot` an absent ledger, `RepairHistory` empty history,
  `SequencePresent` false, `MigrationsStatus` every migration pending).
  Surfaced by CI: a deadline-bounded claim transfer waiting on the
  migration lock burned its budget in the pre-lock probes on a slow
  runner and came back "manifest corrupt" instead of the deadline
  error. Every verdict- or result-bearing existence probe now runs
  through a shared check that refuses to treat a probe under a dead
  context as evidence of absence and surfaces the context error
  instead. A transport failure under a live context still reads as
  absent â€” the gorm migrator API exposes no error channel â€” so this
  narrows the fabrication window to what is detectable.

- **account: `Identity.LastUsedAt` is now `*time.Time` â€” nil means the
  identity has never logged in.** The non-pointer field made every
  `Create` that left it unset write Go's zero time, which go-sql-driver
  hardcodes to `'0000-00-00'` â€” rejected by MySQL 8.4's default strict
  `NO_ZERO_DATE` mode, so any code path creating an Identity without an
  explicit LastUsedAt failed on MySQL (only the OAuth login path, which
  always stamps the moment of linking, happened to dodge it). The column
  was already nullable in all three dialects, so no DDL or baseline
  changes: never-used identities now store SQL NULL, and the JSON body
  omits `last_used_at` instead of emitting a fake
  `0001-01-01T00:00:00Z`.

- **where: `WithCursorBy` no longer treats an empty-string boundary value
  or id 0 as "first page".** Only a nil fieldCursor starts from the top; a
  legitimate empty-string value previously matched the first-page heuristic
  and restarted pagination â€” an infinite loop for the caller.

- **store: `ListWithCursor` accepts filter options only.** A malformed cursor
  direction is now rejected on the first page too â€” it used to be validated
  only on the cursor-carrying branch and silently scanned ascending. Caller
  options that would fight the keyset invariant or the size+1 lookahead
  (`WithOrder`, `WithPage`, `WithLimit`, `WithOffset`, `WithCount`,
  `WithMaxPageSize`, nested cursor options) are rejected as
  `apierr.ErrInvalidArgument`: a stray order took precedence over the keyset
  ordering and silently mis-ordered pages, and a tighter page-size cap
  clipped the lookahead so a page reported "no more rows" (empty NextCursor)
  while rows remained. The guard runs inside the one real, scoped query
  build, so custom `where.Option`s execute exactly once and never see an
  unscoped query. Filter options remain supported. The direct size and
  direction parameters classify the same way: their validation errors now
  map to `apierr.ErrInvalidArgument` too, where the raw `where` sentinel
  previously fell through `store.MapError` and surfaced as a 500.

- **db/store: admin-role lists are immutable and validated at every entry
  point.** `StorePolicy` validation rejects blank `admin_roles` entries, and
  `store.WithAdminRoles` / the deprecated `store.SetDefaultAdminRoles` panic
  on blank role names at configuration time. The handle clones the policy at
  `Open`, `StorePolicy()` returns defensive clones, `WithAdminRoles`
  snapshots its input when the option is created (before validation, so a
  later mutation can neither rewrite the list nor sneak a blank role past
  the check), and `SetDefaultAdminRoles` copies its input â€” mutating a
  caller-held slice can no longer silently rewrite the admin list of stores
  built later, nor race `store.New`.

- **docs: correct the per-store admin-role guidance.** `docs/db.md` Â§11.1
  recommended `WithScope(store.OwnerScope(...))` to declare roles â€” that
  configuration intersects bypass sets rather than overriding them and is now
  called out as a misconfiguration; the documented path is
  `db.store.admin_roles` / `store.WithAdminRoles`.

### Changed

- **store: `ListWithCursor` cursors are opaque, composite and tie-safe.**
  The cursor parameter is now an opaque token â€” pass the empty string for
  the first page and feed `CursorPage.NextCursor` back verbatim â€” instead
  of the previous raw boundary value, and the keyset is composite
  `(field, rid)`: non-unique sort columns such as `created_at` no longer
  silently skip rows sharing a boundary value, with the public RID
  breaking ties so the internal numeric key never rides a client-visible
  token. Tokens bind format version, field and direction and are rejected
  as `apierr.ErrInvalidArgument` when replayed against a different
  contract; the token's value-kind tag is additionally pinned to the
  field's schema-derived kind â€” derived by probing a zero-value row
  through the encoder's exact pipeline (GORM `Field.ValueOf`, which wraps
  serializer fields into Valuers, plus Valuer resolution), so fields whose
  wire type differs from their Go type â€” `gorm.io/datatypes.Time` (int â†’
  string), `serializer:unixtime` (int64 â†’ time.Time) â€” pin to the wire
  kind. Because `driver.Valuer` does not require the concrete wire type to
  stay constant, every actual boundary is revalidated against that pin
  before signing: kind drift or a same-kind value outside the pinned width
  is a server-side field-contract error, never a token the next request
  rejects as forged. Values are range-validated at the field's declared
  width (an int8 field rejects an int64-sized token cleanly instead of
  failing in the driver); NaN is rejected on both sides, time values
  use strict RFC3339 encoding so extended years or sub-minute offsets cannot
  produce an undecodable token, and an invalid-UTF-8 string boundary is
  refused at signing â€” json.Marshal would silently replace it with U+FFFD,
  making the token decode to a DIFFERENT value and scan the next page from
  a wrong position.
  Fields whose token kind cannot be derived statically â€” `sql.Null*`
  wrappers, `[]byte` â€” are rejected up front as a configuration error
  rather than issuing an unconsumable token or skipping forged-kind
  validation. A forged tag can therefore never smuggle a mistyped
  parameter into the row-value comparison. Boundary values round-trip
  with their Go types intact (a stringified timestamp would mis-compare
  against typed columns; time values keep their zone so SQLite's text
  comparison matches the stored form) and the encoder normalises defined
  scalar types, non-nil pointers and `driver.Valuer` wrappers; when the
  size+1 lookahead has proven a next page exists, a NULL or unencodable
  boundary value returns an error instead of silently ending pagination
  with an empty NextCursor. The tie-breaker binds to the model's RID
  column directly â€” stores that never exposed "id" in their query
  allowlist paginate all the same, and a public column alias cannot
  repoint it. An intentional beta contract correction (arch-backlog #2).

- **store: before-update hooks receive the resolved `ChangeSnapshot`.**
  `WithBeforeUpdate` callbacks get the built change set â€” public field names
  mapped to the values about to be written, whole-whitelist `Fields` updates
  fully expanded, recursive-copy accessors â€” instead of the opaque `Changes`
  interface, which exposed no way to inspect what was changing (an
  intentional beta hook-contract correction, mirroring the earlier
  `EntityChanged` snapshot change). Changes are now built before hooks run,
  so static whitelist/protected-column validation precedes user logic on the
  single-row path exactly as it already did for batch writes, and
  `BatchUpdate` builds every item before any hook or SQL â€” inside a
  caller-owned transaction a later item's build error no longer surfaces
  after earlier items' statements ran.

- **store/db: close the reviewed data-layer contract gaps.** Framework-managed
  ID, revision, timestamp, soft-delete and owner columns can no longer be
  reopened by explicit update lists or aliases; every successful ordinary
  update plus soft delete/restore advances the row revision, while Upsert keeps
  its separately documented no-revision limitation. `EntityChanged` now uses
  inspectable `LocatorSnapshot`, `ObjectSnapshot` and `ChangeSnapshot` payloads
  whose accessors return recursive copies (an intentional beta event-contract
  correction). `Fields` now requires the Store's concrete model type and rejects
  shape-compatible DTOs, keeping value extraction and optimistic-lock metadata
  on the same parsed GORM schema. Built-in NULL filters and non-raisable
  pagination caps narrow the trusted custom-`where.Option` surface. Auto
  migration validates every table declaration before DDL and resolves fields
  through GORM schema metadata; migration-lock cleanup is deadline-bounded and
  discards sessions when unlock cannot be confirmed; SQLite constraint mapping
  now distinguishes duplicate keys from NOT NULL/CHECK/FK failures by extended
  result code.

- **db: repair calls carry identity and require a reason everywhere.**
  `RepairSequenceClaim` now demands a non-empty `Reason` (same 1024-character
  rule as ledger repairs) and `chok migrate repair claim` grew a mandatory
  `--reason`. `RepairOptions` and `RepairClaimOptions` gained an optional
  `Operator` â€” explicit values must validate, empty values derive user@host
  best-effort â€” and both reports echo the persisted `Operator` and
  `ChokVersion`. Adding these fields breaks non-keyed struct literals of
  those types; keyed literals stay source-compatible. Least-privilege repair
  accounts additionally need CREATE on first repair, ALTER across engine
  upgrades and INSERT steadily for the history table. `LoadMigrations` now
  rejects filenames containing control characters, path separators or invalid
  UTF-8 â€” the same rule the history reader enforces, so a loadable migration
  can never persist history its own reader rejects.

- **db: owned sequences now require an explicit component owner.**
  `OwnedSequence` accepts `SequenceOwner` (required) and `SequenceVersion`
  (optional). Existing direct call sites remain source-compatible after adding
  the owner option, but the function type changed and assignment to its old
  exact signature no longer compiles; this is an intentional beta-period API
  correction. Built-in kinds are bound to their account/audit/authz package
  owners, and `manifest` is permanently forbidden as a sequence kind.

- **store: truthful upsert events and stricter validation.** `Upsert` and
  `BatchUpsert` require a non-empty conflict target and a non-empty update
  whitelist, and validate static arguments before create hooks. They now
  publish payload-free `OpUpsert` events instead of `OpCreate` objects whose
  hook-generated RID could differ from the persisted conflict row;
  `BatchUpsert` publishes one type-wide invalidation event per call rather than
  per input object. This is a beta-period event-contract correction.

## [2.0.0-beta.5](https://github.com/zynthara/chok/compare/v2.0.0-beta.4...v2.0.0-beta.5) (2026-07-12)

### Added

- **account: `WithPasswordCost` option.** Overrides the bcrypt cost used for
  every password write and the constant-time login padding (default
  `auth.DefaultPasswordCost`). Lower it for tests that hash in tight loops,
  raise it for paranoid deployments. The non-existent-email login padding is
  now computed at the service's own cost instead of a package-global cost-12
  hash, so a tuned cost keeps its timing-oracle defense intact.

- **db/batteries: independent audited migration sequences.** account, audit
  and authz now carry dialect-specific forward-only histories in
  `schema_migrations_chok_<kind>`, with checksum, dirty/fence, dialect and
  repair semantics independent from the application's `schema_migrations`.
  The new `db.Sequence` API exposes Apply/Status/Repair to runtime components
  and tooling. Existing AutoMigrate schemas are adopted only after a complete
  dialect-aware catalog fingerprint matches; partial or older shapes fail
  closed. `chok migrate up --component <kind>` / `--all-owned` support
  least-privilege pre-deploy migration jobs, while status and repair render the
  sequence/ledger/dialect identity explicitly.

- **db/store: explicit read-only database instances.** `db.read_only: true`
  lowers the default `auto` migration mode to `off`, rejects an explicit
  `versioned` mode, and rejects transactions,
  migrations, blessed store writes and GORM write callbacks with
  `db.ErrReadOnly`, and requires `store.WithReadOnly()` at store binding.
  Transaction contexts now carry handle affinity, so a named instance cannot
  silently execute on another instance's transaction. SQLite opens with
  `mode=ro`; PostgreSQL and MySQL install per-connection read-only defaults.
  account, audit and authz fail startup when bound to a read-only default DB.

- **db: runtime Prometheus observability.** When the metrics module is
  assembled, database components export low-cardinality query duration/error,
  connection-pool, SQLite maintenance, and versioned-migration metrics. Named
  databases and SQLite primary/read pools remain distinct. Versioned mode
  separates the migration version embedded in the process from the version
  observed in the database and refreshes applied/pending/dirty state on
  `db.migration_status_interval` (default `30s`; `0` keeps only the initial
  sample).

- **db: configurable slow-query threshold.** Module-managed pools log slow
  queries at `db.slow_threshold` (default `200ms`; `0` disables slow-query
  logs). Library-level `db.Open` remains silent.

- **kernel: component schema ownership metadata.** Components can declare
  their framework-owned database tables through `Descriptor.Schema`. The
  declaration is descriptive metadata rather than an implicit migration
  mechanism; chok's built-in declarations are verified against the tables
  produced by their real kernel migration lifecycle.

- **db/store: app-level store policy.** The `db.store` block (and per
  instance, `db.instances.<name>.store`) sets application-wide defaults
  for every `store.New` over that handle: `strict`,
  `require_principal`, `max_page_size`, `default_page_size`. Explicit
  construction options always win; the new `store.WithoutStrict` /
  `store.WithoutRequirePrincipal` options are the per-store opt-outs.
  A zero-value policy changes nothing.

- **db: audited forward-only migrations.** Versioned migration rows now carry
  normalized SHA-256 checksums, crash-persistent dirty state, start/finish
  timestamps and the last execution error. A temporary compatibility fence
  makes old chok binaries fail closed while dirty state exists; SQLite now
  uses a ledger lease across the marker/DDL transaction boundary. `chok
  migrate status --check` reports pending, dirty, checksum drift, missing,
  unverified, out-of-order, renamed and temporarily fenced migrations. Version-scoped `repair
  retry|mark-applied|accept-drift` actions require an inspected checksum and
  operational reason. Legacy three-column ledgers remain readable and adopt
  a trust-on-first-use checksum baseline on the next `up`.

### Changed

- **Breaking metrics label change:** the five `db_migration_*` /
  `db_migrations_*` gauges add a low-cardinality `sequence` label (`app`,
  `chok_account`, `chok_audit`, `chok_authz`). Dashboards and alerts that
  assumed one migration series per database instance must group or select by
  sequence.

- **db: module-managed query errors are logged.** Database components now
  install the chok GORM logger after opening the otherwise-silent library
  handle. Query errors are logged independently of the slow threshold;
  record-not-found results remain exempt.

- **db: framework table catalog is generated from component ownership.**
  `db.FrameworkTables()` is now the alphabetically sorted, caller-safe copy of
  the built-in `Descriptor.Schema` catalog. It describes potential built-in
  ownership rather than asserting that every table exists in the selected
  database instance; migration status and logs use the same terminology.

- **store/handler: pagination envelopes are same-sourced with the SQL.**
  `where.Config` now produces a `where.PageInfo` (effective page /
  size / offset after clamps; a clamped page-based request recomputes
  its offset in clamped units), `store.Page[T]` carries it as `Meta`,
  and `handler.HandleList` renders the envelope from it instead of
  re-parsing the request. **Breaking:** `Store.ListFromQuery` (and the
  `store.Reader` / `handler.QueryLister` interfaces) returns
  `([]T, int64, where.PageInfo, error)` â€” implementers add the third
  return value.

- **db (sqlite): pure-Go driver + single-writer/parallel-reader
  pools.** The sqlite branch now opens through
  `github.com/glebarez/sqlite` (modernc build â€” no CGO, so chok apps
  cross-compile and run on scratch images out of the box) instead of
  the CGO `gorm.io/driver/sqlite`. File databases run a read/write
  split: writes serialize on one dedicated connection
  (`_txlock=immediate`, fair Go-side queueing â€” SQLite only ever
  admits one writer), reads run in parallel on a WAL-snapshot pool
  routed per gorm callback (`gorm.io/plugin/dbresolver`).
  `db.sqlite.max_open_conns` now sizes the READ pool (default
  `max(4, NumCPU)`); the write side is always one connection. Every
  connection gets `_pragma=foreign_keys(1)` â€” SQLite now enforces the
  referential integrity Postgres always did â€” and
  `_pragma=synchronous(NORMAL)`. The db module runs background
  maintenance for file databases: `PRAGMA wal_checkpoint(TRUNCATE)`
  every `db.sqlite.checkpoint_interval` (default `5m`) and
  `PRAGMA optimize` every `db.sqlite.optimize_interval` (default
  `1h`, plus once at close); `0` disables either.
  **Breaking:** mattn-style DSN parameters in `db.sqlite.path`
  (`_synchronous=`, `_sync=`, `_busy_timeout=`, `_timeout=`,
  `_journal_mode=`, `_journal=`, `_foreign_keys=`, `_fk=`) are
  rejected at startup with a pointer to the `_pragma=name(value)`
  form â€” the pure-Go driver would otherwise silently ignore them.
  `_txlock=` keeps its spelling. Apps that relied on SQLite ignoring
  declared foreign keys must fix their data or disable enforcement
  explicitly (`?_pragma=foreign_keys(0)`).
