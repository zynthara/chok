# Changelog

All notable changes to this project are documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Releases are cut manually since v2: a maintainer writes the entry
here, tags, and pushes — goreleaser publishes the artifacts. Public
API changes must land together with their entry (CI enforces this via
`apidiff` against the latest release tag). For design-level release
notes (the *why* behind each version), see
[`docs/changelog.md`](docs/changelog.md). Entries up to 0.1.4 were
generated by release-please, retired with v2.

## [Unreleased]

### Added

- **store: `Patch(req)` — cast/patch write path for partial updates.**
  A third `Changes` constructor beside `Set` and `Fields` that derives the
  update set from a request DTO's non-nil pointer fields, retiring the
  per-field `if req.X != nil { ... }` dance in update handlers (Ecto
  changeset's *cast*, adapted to Go's pointer-as-optional idiom).
  `Patch(req).Onto(&obj)` applies the values onto an already-loaded model
  and reuses `Fields`' implicit optimistic lock and version write-back;
  bare `Patch(req)` pairs with `WithVersion` like `Set`. Participation is a
  restricted subset of encoding/json field visibility — naming, exportedness
  and embed promotion follow encoding/json (a shallow field excluded from
  patching still shadows a deeper same-name field), then Patch keeps only
  pointer fields (nil = absent, zero values written, `*"" ` clears a column),
  the public name is the json tag's first segment (the Go field name when
  untagged), `store:"-"` opts a field out, named embeds stay nested, and a
  same-depth clash of two patchable fields fails loud. The full DTO shape is validated on
  every build (fields nil this call included), so an unknown, protected or
  type-incompatible field is a 500 on the first request rather than a
  latent trap sprung when a client first sends it; the type rule is strict
  assignability (a nullable `*E` column accepts `E`), no implicit
  conversion. An all-nil call is `ErrEmptyPatch` (mapped to 400); a type
  with no patchable field is `ErrNoPatchableFields` (500). `IsEmpty()`
  lets a handler short-circuit a no-op PATCH without a query. The write
  kernel, interfaces and whitelist machinery are unchanged — the surface
  is purely additive (`Patch`/`PatchChanges` with `Onto`/`NoLock`/
  `IsEmpty`, and the two sentinels), so apidiff sees only additions.
  `examples/blog`'s update handler now uses it.

- **CLI: `chok gen fields` — compile-checked field references.** Scans a
  package for store-tagged models (syntax-level, so regeneration works
  mid-refactor even when the package does not compile) and generates one
  `<Model>Fields` struct var per model in a single `chok_fields_gen.go`,
  committed beside the models. Values are the public field names the
  store whitelists key on — the JSON name, falling back to the GORM
  column for JSON-hidden fields — so
  `where.WithFilter(PostFields.Title, v)` replaces the hand-typed
  `"title"` and a typo'd field name fails the build instead of a request.
  References are stable under `WithColumnAlias` (keys, not columns); the
  base-model trio `ID`/`CreatedAt`/`UpdatedAt` is generated query-only,
  and `version`/protected lifecycle columns get no symbol at all.
  Column-ness is classified syntactically to mirror GORM's parse:
  builtins, byte slices AND fixed byte arrays, exact `driver.Valuer`
  implementers, `GormDataType` types,
  `time.Time`/`sql.Null*`/`gorm.DeletedAt`/the datatypes STORAGE types
  (query-expression types are excluded) and anything proven by a
  `gorm:"type:..."`/serializer tag (or its `gorm:"json"` shorthand)
  generate. Method sets follow Go's real selector rules — shallowest
  and unique wins, so same-depth ambiguity or a shallow wrong-signature
  method/field shadowing a deeper exact one is NOT an implementation;
  aliases inherit (including interface signatures written through
  aliases), defined types do not but keep promotions from their
  underlying struct's embeds — and local defined types resolve through
  their underlying shape with generic arguments substituted, so a
  defined scalar (anonymous embeds included), a defined `time.Time` or
  `Bytes[byte]` is a column while a defined slice is the has-many
  relation it really is; anonymous struct embeds obey GORM's embed rule
  (only a real Valuer, a time-convertible shape, or a GormDataType
  literally returning "time"/"bytes" stays a column — other data-type
  values, serializer and type tags don't prevent expansion, and a
  dynamic GormDataType fails loud); relation-shaped fields are skipped
  with a warning (the runtime ignores their tag too — no dead symbols);
  an undecidable type — unknown cross-package, unsupported `uintptr`, a
  method set the scan cannot fully see — fails loud instead of
  guessing. Verified
  promotions — `store` tags inside embedded structs that GORM lifts at
  runtime, including `gorm:"embedded"` wrappers whose target type is
  unexported (GORM promotes by field name) — warn even when the
  embedding struct has no direct tags of its own, and build constraints
  are honored (go/build MatchFile).
  `--dir` is repeatable, `--check` is the CI drift gate (wired for
  `examples/blog`), and a
  directory whose models lost their last `store` tag has its generated
  file removed. Purely a compile-time shell: no store/where API changed,
  bare strings stay valid forever, and the generator-vs-runtime naming
  contract is pinned by a semantic-latch test that resolves every
  generated value through a real store's whitelists.

- **store: aggregates — `Sum` / `Avg` / `Min` / `Max` / `CountDistinct` /
  `GroupBy`.** The front door for the dashboard reads that previously
  required `Unsafe`, as free functions beside the `Pluck` family and
  under exactly `Count`'s read semantics: fields resolve through the
  query allowlist (an undeclared field stays a raw
  `where.ErrUnknownField` — server code wrote the name, so it is a 500,
  not a client 400), scopes apply fail-closed, soft-deleted rows are
  excluded, and filter options narrow the aggregated set while
  pagination/ordering options are stripped exactly like `Count` strips
  them. Column kinds are schema-derived by the same wire-type probe the
  cursor encoder uses: `Sum`/`Avg` accept numeric columns only,
  `Min`/`Max` additionally accept time columns; string/bool columns are
  not aggregatable (MIN over text is collation-defined per dialect), and
  `CountDistinct` demands a comparable scalar (a PostgreSQL `json`
  column is refused up front — including a Go string field declared
  `gorm:"type:json"` — instead of failing mid-query; string cardinality
  follows the column's collation). On SQLite — which stores timestamps
  as text in the writer's zone and compares them lexicographically —
  aggregates read time columns through a UTC-normalising expression, so
  MIN/MAX pick the chronological (not lexicographic) extreme across
  mixed zone offsets and one instant written under two offsets groups
  and counts as one value, at millisecond precision (matching MySQL's
  DATETIME(3) write precision) and returned in UTC; numeric legacy
  storage reads as Unix seconds via an explicit typeof branch (never
  SQLite's 'auto' heuristic, which misreads early-1970 Unix seconds as
  Julian days). PostgreSQL timestamptz compares instants natively.
  MySQL time columns are DATETIME — naked wall clocks — which chok
  writes on the UTC baseline (see the Changed entry below), so instants
  compare correctly across chok writers regardless of any process TZ;
  what remains is an obligation on non-chok connections writing into
  the same database to use UTC wall clocks too, since a foreign-zone
  write history cannot be repaired at read time. The same bound covers
  ordering, range filters and cursors, not just aggregates. JSON columns are
  refused by the migrator-resolved dialect type as well, so custom
  types mapping to JSON via `GormDBDataType` cannot slip past the
  logical-type check. The capability matrix has two halves: the wire
  kind governs Go-side result convergence, and the REAL database column
  type decides whether the database operation is legal at all. That type
  is read from pure catalog metadata (`pragma_table_info` on SQLite,
  `information_schema` on PostgreSQL/MySQL — never GORM's
  `Migrator.ColumnTypes`, whose implementations sample the data table
  with an unscoped `SELECT * ... LIMIT 1`; resolved lazily on first
  aggregate and cached — not the model-rendered DDL type, which can
  differ from the actual column under `migrate: versioned/off`) and
  matched against an EXACT per-dialect allowlist (not substring, which
  would sweep PostgreSQL `interval`/`int4range`/`integer[]`/`daterange`/
  `time` into the wrong family). An `int64` field over a real `text`
  column fails closed at entry instead of returning a lexicographic MIN,
  and ranges/intervals/arrays/time-of-day and unrecognised column types
  are refused with Unsafe as the escape. Catalog column names are matched
  case-insensitively on SQLite and MySQL (whose identifiers are
  case-insensitive), so a `versioned/off` migration that declared `QTY`
  still resolves against a model's `qty`; the fold is ASCII-only, matching
  how those databases compare identifiers (a full Unicode fold would
  merge the Kelvin sign U+212A onto ASCII `k` and mistype a
  Unicode-distinct column), and PostgreSQL keeps its quoted-identifier
  case. The string family admits `char`/`nchar` (SQLite) and `enum`
  (MySQL) alongside `varchar`/`text`/`uuid`. The Store's scopes are
  applied before the catalog read, so an unauthenticated caller on an
  owned model is rejected (401) purely in memory — it issues no database
  operations, and fails with the auth error rather than a downstream
  catalog/permission error.
  Results converge across the three dialects to caller-declared Go
  types: `Sum[int64]` is exact over integer columns and errors loudly
  past the int64 range (never silent truncation), `Sum[float64]` widens
  with the documented 2^53 trade, `Avg` is always `float64`, and SQL
  NULL (zero rows, or all inputs NULL) surfaces as `ok=false` — never
  conflated with a real zero. `GroupBy[K]` buckets by one allowlisted
  column (`K` must match its wire kind exactly; a NULL group key errors
  loudly and points at `where.WithFilterNotNull` instead of folding into
  the zero-value bucket) and computes positional `AggValue` results
  (`CountRows` / `CountDistinctOf` / `SumOf` / `AvgOf` / `MinOf` /
  `MaxOf`) with kind-checked accessors; groups come back ordered by key
  ascending, and non-filter options — `WithCount` included — are
  rejected: a silently dropped `WithOrder`+`WithLimit` would masquerade
  as top-N. Bool group keys accept only canonical 0/1 storage (a raw 2
  forms its own SQL bucket; folding it onto `true` would hand back
  duplicate keys), and a table-qualified column alias resolves when the
  qualifier names the model's own table. Ordering by
  aggregate values, `LIMIT`, `HAVING`, multi-column grouping and
  string `MIN`/`MAX` are deliberately absent from v1 (see
  `docs/db.md` §5.7); none of the new API touches `Reader` or any
  interface.

### Changed

- **db: MySQL writes times on a UTC baseline.** **Breaking** for
  existing MySQL databases written under the old baseline; a no-op
  only where BOTH of the old baseline's zones were already UTC — the
  process zone (which formatted driver-written values) and the
  session zone (which evaluated `CURRENT_TIMESTAMP`/`NOW()` and
  interpreted TIMESTAMP-column parameters; the server's zone unless
  explicitly set). A UTC process over a non-UTC server still has
  session-written values to rebase. The driver `Loc` is now
  `time.UTC` (previously `time.Local`), so DATETIME columns store the
  UTC wall clock of each instant and parse back in UTC regardless of
  the process TZ or the zone a `time.Time` value carries; and every
  pooled connection pins `time_zone = '+00:00'`, so SQL-evaluated
  timestamps — the `CURRENT_TIMESTAMP` soft delete writes into
  `deleted_at`, `NOW()` in user SQL, TIMESTAMP-column conversion — sit
  on the same baseline instead of forking onto the server's zone.
  Rationale: DATETIME stores a naked wall clock, so every time
  comparison (ordering, range filters, cursor keysets, aggregates)
  compares instants only when all writers share one transition-free
  baseline. The old `time.Local` tied that to the deployment
  environment, folded distinct instants into one wall clock at every
  DST fall-back even within a single process, and let `deleted_at`
  drift onto the server's zone whenever it differed from the process
  zone. With UTC the conversion is injective — chok writers are
  structurally immune, whatever `TZ` says. What remains is an
  obligation on connections chok does not own: non-chok writers to the
  same database must also write UTC wall clocks. API responses
  serialize MySQL-backed timestamps with a `Z` offset now — for
  driver-written DATETIME values that is the same instant in a
  different rendering, but TIMESTAMP values whose internal instants
  were already correct (`DEFAULT CURRENT_TIMESTAMP`-generated) were
  previously RETURNED skewed by (old session − old process): the old
  read parsed the session-zone rendering in the process zone. The
  upgrade corrects their API-visible instants by that difference with
  no data change — clients that stored or cached the old readings see
  these fields move. DATE columns keep their stored civil
  dates (a date has no zone to rebase), but the write contract
  changes: the stored date is now the UTC calendar date of the
  instant — construct date-only values at UTC midnight
  (`time.Date(y, m, d, 0, 0, 0, 0, time.UTC)`); a local midnight east
  of UTC lands on the PREVIOUS UTC day (2026-07-04 00:00 at +08:00
  stores 2026-07-03, where the old baseline stored 2026-07-04), and
  reads come back as UTC midnight of the stored date.
  **Migration** (databases written by ≤ 2.0.0-beta.6): take a backup,
  stop EVERY writer to the database — all old chok instances AND
  every non-chok write source: external services, ETL and scheduled
  jobs, ad-hoc operator SQL (the same foreign writers this entry
  obliges to use UTC). The recipe converts a frozen snapshot:
  anything written during the window, old-baseline or new, either
  escapes conversion or gets converted twice. The window admits no
  rolling upgrade: an old instance writing after the rebase leaves
  unconverted rows, and a new instance booting before it triggers the
  already-booted contingency below. Note that the NEW binary's
  `chok migrate` WRITE commands count as that first boot too: bare
  `up` writes new-baseline rows into the application ledger,
  `up --all-owned` / `--component` additionally refreshes the
  manifest, and `repair mark-applied` rewrites a ledger row's
  `applied_at`; `status` stays read-only and is safe inside the
  window. Then rebase
  per column BY PROVENANCE — the old baseline had two zones, so one
  blanket conversion is wrong whenever they differed. Verify what each zone actually was: Go resolved the
  process zone from `TZ`, else `/etc/localtime`, else UTC (a minimal
  image with neither ran UTC; an image baking a zone into
  `/etc/localtime` did not); the session zone was the server's
  `time_zone`/SYSTEM unless the app pinned one.

  1. Driver-written DATETIME columns (gorm-managed `created_at` /
     `updated_at`, business `time.Time` fields): rebase from the old
     PROCESS zone.
  2. SQL-evaluated DATETIME columns (chok's soft-delete `deleted_at`
     — written by `CURRENT_TIMESTAMP` — and any column your own SQL
     fed with `NOW()`): rebase from the old SESSION zone.
  3. Parameter-written TIMESTAMP columns (chok's migration-ledger
     `applied_at` / `started_at` / `finished_at` and manifest
     `claimed_at` / `updated_at` among them): the stored instant is
     correct only where the two old zones matched; where they
     differed it is skewed by exactly their difference — the old
     driver formatted the process-zone wall clock and the old session
     interpreted it, an error the old asymmetric read canceled and
     the new symmetric UTC read exposes. Rebase from the old process
     zone TO the old session zone (under the new pinned session that
     lands the true instant). Values generated purely by
     `DEFAULT` / `ON UPDATE CURRENT_TIMESTAMP` carry correct instants
     and need nothing — and that split is PER ROW, not per column: a
     column that mixes both provenances must convert only its
     parameter-written rows, because the blanket conversion moves
     every correct DEFAULT instant by the same difference it repairs
     the skewed ones with. chok's own migration ledger acquired
     exactly that mix — 2.0.0-beta.4 and earlier filled `applied_at`
     by DEFAULT, beta.5+ writes it by parameter — and the DEFAULT-era
     rows are identifiable: the beta.5 ledger adoption stamped them
     `provenance` `legacy` or `checksum-tofu` (their `finished_at`
     was backfilled from `applied_at`, so it is DEFAULT-derived too).
     Run EVERY rebase statement BEFORE the new version first boots,
     inside the stop-writers window: the first boot applies pending
     migrations and refreshes the manifest's `updated_at`, and those
     writes land correct UTC values carrying the very
     `applied`/`baseline` marks (and manifest columns) the statements
     convert — rebasing after boot drags the new, correct values off
     baseline by the old zone difference. Pre-boot execution is safe
     on any ledger that ever ran beta.5+, because the old engine
     stamped `applied`/`baseline` at insert time (the boot-time
     adoption only renames the remaining empty-provenance rows to
     `legacy`/`checksum-tofu` later), and the positive predicate
     skips those empty beta.4-era rows — DEFAULT fills that need no
     conversion anyway. A ledger coming straight from beta.4 lacks
     these columns entirely: the statement fails loud, and that IS
     the answer — a pure beta.4 ledger was all DEFAULT-filled, skip
     it. If the new version already booted before you rebased, do
     not run the blanket statements: bound the ledger conversion to
     pre-upgrade rows (`AND version <= <highest version applied
     before the upgrade>`), and for the manifest convert `claimed_at`
     ONLY for kinds that already existed before the upgrade (`AND
     kind IN (...)` — a row first claimed or adopted at that boot
     carries a new-baseline `claimed_at`, and no data-side marker
     separates the two: `provenance` says adopted/claimed but not
     when, and the boot refreshes `chok_version` on every row; the
     pre-upgrade kind set is something only you know). Skip
     `updated_at` — it was just refreshed on the new baseline. If the
     new binary also ran `chok migrate repair` after that boot, two
     more exclusions apply: `mark-applied` rewrote an old version's
     `applied_at`/`finished_at` on the NEW baseline, inside the
     version bound — subtract the versions you repaired from the
     bounded statement; and every ledger repair, plus a manifest
     claim transfer, appended a repair-history row with a
     new-baseline `repaired_at` — bound that statement to pre-upgrade
     rows (`AND id <= <pre-upgrade MAX(id)>`) instead of
     whole-column. Wherever a boundary is unknown, skip the
     bookkeeping table entirely: its skew misreports history, nothing
     else. Business tables get no such pass, and an id bound is only
     HALF a line: `AND id <= <the table's pre-upgrade MAX(id)>`
     excludes post-boot INSERTS, but a column the new version can
     REWRITE on a pre-upgrade row — gorm's autoUpdateTime
     `updated_at`, a post-boot soft delete's `deleted_at`,
     app-refreshed fields like a login's `last_used_at` — changes
     value without changing id and carries no data-side marker
     separating refreshed from stale (the same property as the
     manifest's `updated_at` above). Bound insert-shaped columns
     (`created_at`, immutable event stamps) by id; for refreshable
     columns that post-boot traffic may have touched, no in-place
     conversion is correct — restore the backup and redo the cutover
     in order.
     The manifest table has been parameter-written since it exists,
     so in the pre-boot path its whole columns convert. One honest tail: a row later
     touched by `chok migrate repair` had `applied_at` rewritten by
     parameter regardless of its provenance — if repair ran before
     this upgrade, classify those versions by hand. For your own
     mixed columns, partition by whatever marker records the writer;
     without one, the two hypotheses cannot be told apart from the
     data alone (the same no-zone-stored limit). The framework ledger
     columns are bookkeeping plus a transient lease CAS — a residual
     skew misreports history but does not affect the engine; rebase
     them for clean forensics.

  ```sql
  -- enumerate the DATETIME columns of the current schema
  SELECT table_name, column_name FROM information_schema.columns
  WHERE table_schema = DATABASE() AND data_type = 'datetime';
  -- 1. driver-written DATETIME: old process zone → UTC (fixed
  --    offsets convert exactly and need no server tz tables)
  UPDATE `<table>`
  SET `<col>` = CONVERT_TZ(`<col>`, '<OLD_PROCESS>', '+00:00')
  WHERE `<col>` IS NOT NULL;
  -- 2. SQL-evaluated DATETIME (deleted_at, NOW()-fed columns):
  --    old session zone → UTC
  UPDATE `<table>`
  SET `deleted_at` = CONVERT_TZ(`deleted_at`, '<OLD_SESSION>', '+00:00')
  WHERE `deleted_at` IS NOT NULL;
  -- 3. parameter-written TIMESTAMP ROWS, only where the two old
  --    zones differed; the predicate must exclude DEFAULT-written
  --    rows when the column mixes provenances. TIMESTAMP columns
  --    need their own enumeration — the DATETIME listing above does
  --    not surface them. chok's own framework tables appear in this
  --    enumeration too: they are handled by the dedicated statements
  --    below, so EXCLUDE them from the generic sweep — a second pass
  --    overshoots the true instant and recreates the same-size skew
  --    in the opposite direction:
  SELECT table_name, column_name FROM information_schema.columns
  WHERE table_schema = DATABASE() AND data_type = 'timestamp';
  UPDATE `<table>`
  SET `<ts_col>` = CONVERT_TZ(`<ts_col>`, '<OLD_PROCESS>', '<OLD_SESSION>')
  WHERE `<ts_col>` IS NOT NULL AND <parameter-written rows only>;
  -- chok's migration ledgers: schema_migrations plus every
  -- owned-sequence ledger schema_migrations_chok_<kind> — the
  -- built-in batteries and any custom or third-party sequence
  -- registered through db.OwnedSequence — EXCEPT the two reserved
  -- kinds below, which have their own shapes. Run BEFORE the new
  -- version first boots (see above): only 'applied'/'baseline' rows
  -- are parameter-written, stamped by the old engine at insert time.
  -- The version=0 compatibility fence keeps an empty provenance with
  -- a parameter-written applied_at — engine bookkeeping, safe to
  -- leave unconverted.
  UPDATE `schema_migrations`
  SET applied_at  = CONVERT_TZ(applied_at,  '<OLD_PROCESS>', '<OLD_SESSION>'),
      started_at  = CONVERT_TZ(started_at,  '<OLD_PROCESS>', '<OLD_SESSION>'),
      finished_at = CONVERT_TZ(finished_at, '<OLD_PROCESS>', '<OLD_SESSION>')
  WHERE provenance IN ('applied', 'baseline');
  -- reserved kind 1: the manifest (schema_migrations_chok_manifest)
  -- is all parameter-written — whole columns convert. For this and
  -- the repair history below, an absent table just means there are
  -- no rows to convert: the statement fails loud with
  -- "table doesn't exist" and changes nothing — skip it. Do not
  -- infer history from absence (the repair history, for one, is
  -- created lazily by the first repair, owned or not).
  UPDATE `schema_migrations_chok_manifest`
  SET claimed_at = CONVERT_TZ(claimed_at, '<OLD_PROCESS>', '<OLD_SESSION>'),
      updated_at = CONVERT_TZ(updated_at, '<OLD_PROCESS>', '<OLD_SESSION>');
  -- reserved kind 2: the repair history
  -- (schema_migrations_chok_repairs) has no provenance column; its
  -- repaired_at is parameter-written on every row — whole column
  -- converts in the pre-boot path (already booted: bound by
  -- pre-upgrade MAX(id), see the contingency above):
  UPDATE `schema_migrations_chok_repairs`
  SET repaired_at = CONVERT_TZ(repaired_at, '<OLD_PROCESS>', '<OLD_SESSION>');
  ```

  Caveats: a column declared `ON UPDATE CURRENT_TIMESTAMP` is
  auto-bumped by any UPDATE that does not assign it explicitly —
  rebase it in the same statement as its siblings (an explicit
  assignment suppresses the auto-bump); a stored generated DATETIME
  column appears in the enumeration but cannot be assigned — skip it
  and let it recompute from its rebased inputs. A history written
  under a DST zone needs the named zone instead
  (`CONVERT_TZ(col, 'America/New_York', ...)`, requires the server's
  tz tables) and remains ambiguous for wall clocks inside fall-back
  folds — those cannot be recovered by any tool; that unfixability is
  part of why the baseline moved. The executable form of this recipe
  — split zones, all three provenances, a TIMESTAMP column mixing
  parameter-written and DEFAULT-written rows, and the read-side
  correction — is pinned by `TestMySQLUTCBaseline_LegacyRebaseRecipe`.

## [2.0.0-beta.6](https://github.com/zynthara/chok/compare/v2.0.0-beta.5...v2.0.0-beta.6) (2026-07-17)

### Added

- **store: declarative constraint→field mapping for duplicate errors.**
  `store.WithConstraintFields(map[string]string)` declares which public
  field a unique constraint should blame (Ecto's `unique_constraint`
  idea). A declared hit populates the new `DuplicateEntryError.Field`,
  and `store.MapError` then reports response metadata `field` instead
  of the raw constraint name — index names leak schema layout and drift
  with migrations, field names are the API's own vocabulary. Undeclared
  constraints keep the existing `constraint` metadata unchanged. Keys
  match the driver-reported identifier with table qualifiers stripped
  (MySQL 8 reports `table.key`); SQLite names no index — it reports the
  violated column list (`delete_token` included for `SoftUnique`
  indexes) — so cross-dialect declarations list both spellings.

- **store: `ListIn` — the auto-chunked second half of the two-step IN
  pattern.** `store.ListIn(ctx, s, field, values, opts...)` retrieves
  every row whose field value is in the set, deduplicating the values
  and splitting them into IN lists of at most `where.MaxInList`
  automatically — the blessed cross-table pattern no longer hands the
  chunk loop to the caller. The anchor is one big IN's SET semantics:
  when the set spans multiple chunks, combined rows are additionally
  deduplicated by primary key, because database equality can be wider
  than Go equality (a case-insensitive collation matches one row from
  values in two different chunks). Each chunk runs under exactly
  List's read path (query allowlist, scopes, soft-delete rules), and
  an empty value set still runs one degenerate no-match query so the
  allowlist, the option guard and fail-closed scopes all apply —
  never a silently successful empty page. Options are filters-only
  (ordering, pagination, count and page-size caps do not compose
  across chunks and are rejected), the result carries no order
  guarantee, and — like `ListByIDs` — it is server-side plumbing sized
  by the value set, deliberately not subject to the Store's
  max-page-size cap (a per-chunk clamp would silently drop rows
  mid-set). Multiple chunks are multiple SELECTs, not one statement,
  and a transaction restores single-snapshot semantics only where the
  default isolation provides a transaction-wide snapshot (`Store.Tx` /
  `db.RunInTx` use the database default): SQLite and MySQL/InnoDB
  qualify; PostgreSQL's default READ COMMITTED re-snapshots per
  statement — escalate to REPEATABLE READ yourself (via the Unsafe
  door, first statement in the transaction) when cross-chunk snapshot
  consistency matters there.

- **store: pessimistic locking via `GetForUpdate`.** The transactional
  counterpart to Update's optimistic locking:
  `GetForUpdate(ctx, locator, opts...)` reads a row under
  `SELECT ... FOR UPDATE`, holding off concurrent lockers and writers of
  that row until the enclosing transaction commits. The entry point is
  verifiable by construction: it requires a transaction on the store's
  own handle (`ErrLockRequiresTx` otherwise — a lock under autocommit is
  released before the caller can use it), refuses read-only stores
  (`db.ErrReadOnly`) and rejects `WithPreload` (`ErrLockPreload` —
  association queries run outside the lock). On SQLite the driver drops
  the locking clause, but chok's SQLite shape serializes every
  transaction and write onto a single write connection (file databases
  pin a one-connection write pool, `_txlock=immediate` by default;
  memory databases run on one pinned connection), so the enclosing
  transaction excludes every concurrent writer — stronger than a row
  lock — and the observable guarantee is uniform across dialects.

- **where: `WithCursorByField`.** The composite-cursor option whose
  tie-breaker is a caller-chosen column: the sort field resolves through
  the query allowlist (client-facing input), while the tie column is
  identifier-validated but deliberately not allowlist-resolved — it is
  trusted infrastructure, which is how `ListWithCursor` binds the model's
  RID column without requiring "id" in the allowlist. `WithCursorBy`
  (internal id tie-breaker) remains the server-side DSL tool.

- **store: unified admin-role policy.** `db.store.admin_roles` (and
  `db.instances.<name>.store.admin_roles`) plus the per-store
  `store.WithAdminRoles(...)` option name the principal roles that bypass the
  automatic OwnerScope AND may set OwnerID explicitly on create. One list —
  resolved as option > policy > deprecated package default and captured at
  store construction — now drives both sides, closing two authorization gaps:
  the write-side owner fill used to re-read the mutable global list per call
  (so `SetDefaultAdminRoles` after construction skewed write semantics away
  from the already-frozen query scope), and the previously documented
  per-store override (`WithScope(OwnerScope("x"))`) actually intersected
  bypass sets instead of replacing them because scopes compose by AND.
  `WithAdminRoles()` with no arguments removes every bypass (fail-closed);
  `SetDefaultAdminRoles` remains as the deprecated fallback default.

- **db: append-only migration repair history.** Every repair — application
  and owned-sequence `retry` / `mark-applied` / `accept-drift`, and manifest
  claim transfers — now persists its full evidence (action, checksums or
  owners, mandatory reason, operator, chok version) to
  `schema_migrations_chok_repairs` in the repair's own transaction: a history
  row means the business-state compare-and-swap committed, and a repair whose
  history cannot be recorded fails instead of committing. `db.RepairHistory`
  reads the evidence most-recent-first with kind filtering and clamped
  limits; `app` and `repairs` join `manifest` as permanently forbidden
  sequence kinds, and `OwnedSequence` now resolves kind legality through the
  same `ValidateSequenceKind` gate as every other consumer.

- **db: third-party owned-sequence manifest and claim protocol.**
  `db.ManifestEntries`, `db.LedgerSnapshot` and `db.RepairSequenceClaim` expose
  a read-only catalog, file-independent ledger health and lock-protected owner
  transfer, and `db.ValidateSequenceKind` is the shared gate tooling uses
  before deriving ledger identifiers from externally observed kind strings. Every owned sequence persists its component-package owner,
  compatibility engine floor and informational component/chok versions in
  `schema_migrations_chok_manifest`. Apply, repair and claim transfer authorize
  under the migration lock; legacy ledgers are adopted only after validation
  and before any subsequent ledger or schema write. `chok migrate status`
  discovers both claimed and legacy-unclaimed ledgers, while strict `--check`
  fails on unclaimed or file-unverified third-party content unless
  `--ledger-health-only` is explicitly selected.

- **audit/authz: exported auto-schema helpers.** `audit.MigrateSchema` and
  `authz.MigrateSchema` expose each battery's sanctioned AutoMigrate path —
  the exact calls their modules make under `migrate: auto` — mirroring the
  existing `account.MigrateSchema`. Kernel-less embedders and the migration
  behavior fixtures now share one schema source instead of copying raw
  AutoMigrate calls that could drift from the module.

- **store: per-row batch update and atomic batch upsert.** `BatchUpdate`
  applies `Fields` semantics, optimistic locking and scopes to per-object
  payloads in one transaction, restoring framework-mutated versions when the
  batch fails. `BatchUpsert` validates conflict/update columns and duplicate
  conflict tuples before SQL, then executes all GORM chunks atomically. The
  opt-in `BatchWriter[T]` interface exposes both without expanding the existing
  narrow `Writer[T]` contract.

### Changed

- **store: unknown-field errors are classified by entry point.**
  **Breaking:** the programmatic read entry points — `List`, `ListQ`,
  `Count`, `Pluck`/`PluckDistinct`/`PluckIDs`, `ListIn`,
  `ListWithCursor`'s cursor field, and the locator-based `Get` /
  `GetForUpdate` / `Update` / `Delete` / `Restore` / `Exists` — now
  return `where.ErrUnknownField` raw (chain intact, `errors.Is` works;
  `store.MapError` does not map it → 500) instead of pre-mapping it to
  `apierr.ErrInvalidArgument` (400). Field names on these entry points
  are written by server code, so `List(ctx, where.WithFilter("typo",
  v))` is a programming bug that must alarm monitoring — the old
  mapping disguised it as a client mistake. The `ListFromQuery` chain
  keeps the 400 mapping end to end: there the field names arrive in
  the URL and genuinely are client input. VALUE errors
  (`where.ErrInvalidParam` — pages, sizes, cursor tokens, filter
  values) keep their 400 mapping on every entry point, so handlers
  may still pass client pagination values straight into `WithPage` /
  `WithLimit` / cursor size. Migration: route client-supplied FIELD
  names through `ListFromQuery` (or validate them against the
  allowlist in the handler) instead of splicing them into
  `WithFilter` / `WithOrder`; code that matched the old 400 via
  `errors.Is(err, apierr.ErrInvalidArgument)` on programmatic calls
  should match `errors.Is(err, where.ErrUnknownField)`. The update
  side is unchanged (`ErrUnknownUpdateField` was already a 500).

- **store: cursor size discipline.** Two documented bounds join the
  opaque-cursor contract as exported constants: `MaxCursorTokenLen`
  (4096 bytes) caps the token `ListWithCursor` accepts — over-limit
  tokens are rejected as `apierr.ErrInvalidArgument` before any
  base64/JSON work, so token length can no longer buy proportional
  server-side decode allocation — and `MaxCursorValueLen` (1024 bytes)
  caps the string representation of a boundary value the encoder will
  sign. A page whose proven-to-exist next boundary exceeds the value
  bound (or whose assembled token would exceed the token bound — JSON
  escaping can inflate control-character-heavy strings past it) returns
  a server-side error instead of signing a token the framework itself
  refuses to decode, and never silently truncates the value. Previously
  both sides were unbounded. Cursor fields are short scalar keys; if a
  column can legitimately exceed 1KB, it should not key a cursor.

- **store/handler: `ListFromQuery` returns `*Page[T]` and leaves the
  `Reader` contract.** The list envelope now lives in the query layer as
  `where.Page[T]` — `store.Page` stays as an alias — so
  `handler.QueryLister` speaks the same envelope without handler
  importing store. **Breaking:** `Store.ListFromQuery` (and
  `handler.QueryLister`) returns `(*Page[T], error)` instead of
  `([]T, int64, where.PageInfo, error)`; callers read `page.Items` /
  `page.Total` / `page.Meta`. **Breaking:** `store.Reader` no longer
  includes `ListFromQuery` — the read contract is transport-free;
  parsing URL input belongs to the edges (`handler.HandleList`, or the
  `*Store.ListFromQuery` sugar, which both keep working). Nothing
  changes at the HTTP surface.

- **store: `Writer` is the single-row write surface.** The
  Writer/BatchWriter split now follows a single line — single-row
  versus multi-row — instead of its historical accretion. `Upsert`
  (single row) stays on `Writer`. **Breaking:** `BatchCreate` leaves
  `Writer` (and therefore `ReadWriter`); it remains on `BatchWriter`
  and on `*Store`. Code that batch-creates through a `Writer[T]`
  dependency declares `BatchWriter[T]` (or keeps `*Store[T]`).

### Fixed

- **db: table-existence probes no longer fabricate verdicts or results
  out of failed queries.** gorm's `Migrator().HasTable` swallows query
  errors into `false`, so a probe running under an expired or cancelled
  context read as "table absent" — which the migration surfaces
  escalated into fabricated verdicts (`RepairSequenceClaim` reported
  `ErrSequenceManifestCorrupt` "claim exists without ledger" /
  `ErrSequenceUnclaimed`; `SchemaFingerprint` reported a missing table)
  or result-shaped lies (`ManifestEntries` returned an empty manifest,
  `LedgerSnapshot` an absent ledger, `RepairHistory` empty history,
  `SequencePresent` false, `MigrationsStatus` every migration pending).
  Surfaced by CI: a deadline-bounded claim transfer waiting on the
  migration lock burned its budget in the pre-lock probes on a slow
  runner and came back "manifest corrupt" instead of the deadline
  error. Every verdict- or result-bearing existence probe now runs
  through a shared check that refuses to treat a probe under a dead
  context as evidence of absence and surfaces the context error
  instead. A transport failure under a live context still reads as
  absent — the gorm migrator API exposes no error channel — so this
  narrows the fabrication window to what is detectable.

- **account: `Identity.LastUsedAt` is now `*time.Time` — nil means the
  identity has never logged in.** The non-pointer field made every
  `Create` that left it unset write Go's zero time, which go-sql-driver
  hardcodes to `'0000-00-00'` — rejected by MySQL 8.4's default strict
  `NO_ZERO_DATE` mode, so any code path creating an Identity without an
  explicit LastUsedAt failed on MySQL (only the OAuth login path, which
  always stamps the moment of linking, happened to dodge it). The column
  was already nullable in all three dialects, so no DDL or baseline
  changes: never-used identities now store SQL NULL, and the JSON body
  omits `last_used_at` instead of emitting a fake
  `0001-01-01T00:00:00Z`.

- **where: `WithCursorBy` no longer treats an empty-string boundary value
  or id 0 as "first page".** Only a nil fieldCursor starts from the top; a
  legitimate empty-string value previously matched the first-page heuristic
  and restarted pagination — an infinite loop for the caller.

- **store: `ListWithCursor` accepts filter options only.** A malformed cursor
  direction is now rejected on the first page too — it used to be validated
  only on the cursor-carrying branch and silently scanned ascending. Caller
  options that would fight the keyset invariant or the size+1 lookahead
  (`WithOrder`, `WithPage`, `WithLimit`, `WithOffset`, `WithCount`,
  `WithMaxPageSize`, nested cursor options) are rejected as
  `apierr.ErrInvalidArgument`: a stray order took precedence over the keyset
  ordering and silently mis-ordered pages, and a tighter page-size cap
  clipped the lookahead so a page reported "no more rows" (empty NextCursor)
  while rows remained. The guard runs inside the one real, scoped query
  build, so custom `where.Option`s execute exactly once and never see an
  unscoped query. Filter options remain supported. The direct size and
  direction parameters classify the same way: their validation errors now
  map to `apierr.ErrInvalidArgument` too, where the raw `where` sentinel
  previously fell through `store.MapError` and surfaced as a 500.

- **db/store: admin-role lists are immutable and validated at every entry
  point.** `StorePolicy` validation rejects blank `admin_roles` entries, and
  `store.WithAdminRoles` / the deprecated `store.SetDefaultAdminRoles` panic
  on blank role names at configuration time. The handle clones the policy at
  `Open`, `StorePolicy()` returns defensive clones, `WithAdminRoles`
  snapshots its input when the option is created (before validation, so a
  later mutation can neither rewrite the list nor sneak a blank role past
  the check), and `SetDefaultAdminRoles` copies its input — mutating a
  caller-held slice can no longer silently rewrite the admin list of stores
  built later, nor race `store.New`.

- **docs: correct the per-store admin-role guidance.** `docs/db.md` §11.1
  recommended `WithScope(store.OwnerScope(...))` to declare roles — that
  configuration intersects bypass sets rather than overriding them and is now
  called out as a misconfiguration; the documented path is
  `db.store.admin_roles` / `store.WithAdminRoles`.

### Changed

- **store: `ListWithCursor` cursors are opaque, composite and tie-safe.**
  The cursor parameter is now an opaque token — pass the empty string for
  the first page and feed `CursorPage.NextCursor` back verbatim — instead
  of the previous raw boundary value, and the keyset is composite
  `(field, rid)`: non-unique sort columns such as `created_at` no longer
  silently skip rows sharing a boundary value, with the public RID
  breaking ties so the internal numeric key never rides a client-visible
  token. Tokens bind format version, field and direction and are rejected
  as `apierr.ErrInvalidArgument` when replayed against a different
  contract; the token's value-kind tag is additionally pinned to the
  field's schema-derived kind — derived by probing a zero-value row
  through the encoder's exact pipeline (GORM `Field.ValueOf`, which wraps
  serializer fields into Valuers, plus Valuer resolution), so fields whose
  wire type differs from their Go type — `gorm.io/datatypes.Time` (int →
  string), `serializer:unixtime` (int64 → time.Time) — pin to the wire
  kind. Because `driver.Valuer` does not require the concrete wire type to
  stay constant, every actual boundary is revalidated against that pin
  before signing: kind drift or a same-kind value outside the pinned width
  is a server-side field-contract error, never a token the next request
  rejects as forged. Values are range-validated at the field's declared
  width (an int8 field rejects an int64-sized token cleanly instead of
  failing in the driver); NaN is rejected on both sides, time values
  use strict RFC3339 encoding so extended years or sub-minute offsets cannot
  produce an undecodable token, and an invalid-UTF-8 string boundary is
  refused at signing — json.Marshal would silently replace it with U+FFFD,
  making the token decode to a DIFFERENT value and scan the next page from
  a wrong position.
  Fields whose token kind cannot be derived statically — `sql.Null*`
  wrappers, `[]byte` — are rejected up front as a configuration error
  rather than issuing an unconsumable token or skipping forged-kind
  validation. A forged tag can therefore never smuggle a mistyped
  parameter into the row-value comparison. Boundary values round-trip
  with their Go types intact (a stringified timestamp would mis-compare
  against typed columns; time values keep their zone so SQLite's text
  comparison matches the stored form) and the encoder normalises defined
  scalar types, non-nil pointers and `driver.Valuer` wrappers; when the
  size+1 lookahead has proven a next page exists, a NULL or unencodable
  boundary value returns an error instead of silently ending pagination
  with an empty NextCursor. The tie-breaker binds to the model's RID
  column directly — stores that never exposed "id" in their query
  allowlist paginate all the same, and a public column alias cannot
  repoint it. An intentional beta contract correction (arch-backlog #2).

- **store: before-update hooks receive the resolved `ChangeSnapshot`.**
  `WithBeforeUpdate` callbacks get the built change set — public field names
  mapped to the values about to be written, whole-whitelist `Fields` updates
  fully expanded, recursive-copy accessors — instead of the opaque `Changes`
  interface, which exposed no way to inspect what was changing (an
  intentional beta hook-contract correction, mirroring the earlier
  `EntityChanged` snapshot change). Changes are now built before hooks run,
  so static whitelist/protected-column validation precedes user logic on the
  single-row path exactly as it already did for batch writes, and
  `BatchUpdate` builds every item before any hook or SQL — inside a
  caller-owned transaction a later item's build error no longer surfaces
  after earlier items' statements ran.

- **store/db: close the reviewed data-layer contract gaps.** Framework-managed
  ID, revision, timestamp, soft-delete and owner columns can no longer be
  reopened by explicit update lists or aliases; every successful ordinary
  update plus soft delete/restore advances the row revision, while Upsert keeps
  its separately documented no-revision limitation. `EntityChanged` now uses
  inspectable `LocatorSnapshot`, `ObjectSnapshot` and `ChangeSnapshot` payloads
  whose accessors return recursive copies (an intentional beta event-contract
  correction). `Fields` now requires the Store's concrete model type and rejects
  shape-compatible DTOs, keeping value extraction and optimistic-lock metadata
  on the same parsed GORM schema. Built-in NULL filters and non-raisable
  pagination caps narrow the trusted custom-`where.Option` surface. Auto
  migration validates every table declaration before DDL and resolves fields
  through GORM schema metadata; migration-lock cleanup is deadline-bounded and
  discards sessions when unlock cannot be confirmed; SQLite constraint mapping
  now distinguishes duplicate keys from NOT NULL/CHECK/FK failures by extended
  result code.

- **db: repair calls carry identity and require a reason everywhere.**
  `RepairSequenceClaim` now demands a non-empty `Reason` (same 1024-character
  rule as ledger repairs) and `chok migrate repair claim` grew a mandatory
  `--reason`. `RepairOptions` and `RepairClaimOptions` gained an optional
  `Operator` — explicit values must validate, empty values derive user@host
  best-effort — and both reports echo the persisted `Operator` and
  `ChokVersion`. Adding these fields breaks non-keyed struct literals of
  those types; keyed literals stay source-compatible. Least-privilege repair
  accounts additionally need CREATE on first repair, ALTER across engine
  upgrades and INSERT steadily for the history table. `LoadMigrations` now
  rejects filenames containing control characters, path separators or invalid
  UTF-8 — the same rule the history reader enforces, so a loadable migration
  can never persist history its own reader rejects.

- **db: owned sequences now require an explicit component owner.**
  `OwnedSequence` accepts `SequenceOwner` (required) and `SequenceVersion`
  (optional). Existing direct call sites remain source-compatible after adding
  the owner option, but the function type changed and assignment to its old
  exact signature no longer compiles; this is an intentional beta-period API
  correction. Built-in kinds are bound to their account/audit/authz package
  owners, and `manifest` is permanently forbidden as a sequence kind.

- **store: truthful upsert events and stricter validation.** `Upsert` and
  `BatchUpsert` require a non-empty conflict target and a non-empty update
  whitelist, and validate static arguments before create hooks. They now
  publish payload-free `OpUpsert` events instead of `OpCreate` objects whose
  hook-generated RID could differ from the persisted conflict row;
  `BatchUpsert` publishes one type-wide invalidation event per call rather than
  per input object. This is a beta-period event-contract correction.

## [2.0.0-beta.5](https://github.com/zynthara/chok/compare/v2.0.0-beta.4...v2.0.0-beta.5) (2026-07-12)

### Added

- **account: `WithPasswordCost` option.** Overrides the bcrypt cost used for
  every password write and the constant-time login padding (default
  `auth.DefaultPasswordCost`). Lower it for tests that hash in tight loops,
  raise it for paranoid deployments. The non-existent-email login padding is
  now computed at the service's own cost instead of a package-global cost-12
  hash, so a tuned cost keeps its timing-oracle defense intact.

- **db/batteries: independent audited migration sequences.** account, audit
  and authz now carry dialect-specific forward-only histories in
  `schema_migrations_chok_<kind>`, with checksum, dirty/fence, dialect and
  repair semantics independent from the application's `schema_migrations`.
  The new `db.Sequence` API exposes Apply/Status/Repair to runtime components
  and tooling. Existing AutoMigrate schemas are adopted only after a complete
  dialect-aware catalog fingerprint matches; partial or older shapes fail
  closed. `chok migrate up --component <kind>` / `--all-owned` support
  least-privilege pre-deploy migration jobs, while status and repair render the
  sequence/ledger/dialect identity explicitly.

- **db/store: explicit read-only database instances.** `db.read_only: true`
  lowers the default `auto` migration mode to `off`, rejects an explicit
  `versioned` mode, and rejects transactions,
  migrations, blessed store writes and GORM write callbacks with
  `db.ErrReadOnly`, and requires `store.WithReadOnly()` at store binding.
  Transaction contexts now carry handle affinity, so a named instance cannot
  silently execute on another instance's transaction. SQLite opens with
  `mode=ro`; PostgreSQL and MySQL install per-connection read-only defaults.
  account, audit and authz fail startup when bound to a read-only default DB.

- **db: runtime Prometheus observability.** When the metrics module is
  assembled, database components export low-cardinality query duration/error,
  connection-pool, SQLite maintenance, and versioned-migration metrics. Named
  databases and SQLite primary/read pools remain distinct. Versioned mode
  separates the migration version embedded in the process from the version
  observed in the database and refreshes applied/pending/dirty state on
  `db.migration_status_interval` (default `30s`; `0` keeps only the initial
  sample).

  queries at `db.slow_threshold` (default `200ms`; `0` disables slow-query
  logs). Library-level `db.Open` remains silent.

- **kernel: component schema ownership metadata.** Components can declare
  their framework-owned database tables through `Descriptor.Schema`. The
  declaration is descriptive metadata rather than an implicit migration
  mechanism; chok's built-in declarations are verified against the tables
  produced by their real kernel migration lifecycle.

- **db/store: app-level store policy.** The `db.store` block (and per
  instance, `db.instances.<name>.store`) sets application-wide defaults
  for every `store.New` over that handle: `strict`,
  `require_principal`, `max_page_size`, `default_page_size`. Explicit
  construction options always win; the new `store.WithoutStrict` /
  `store.WithoutRequirePrincipal` options are the per-store opt-outs.
  A zero-value policy changes nothing.

- **db: audited forward-only migrations.** Versioned migration rows now carry
  normalized SHA-256 checksums, crash-persistent dirty state, start/finish
  timestamps and the last execution error. A temporary compatibility fence
  makes old chok binaries fail closed while dirty state exists; SQLite now
  uses a ledger lease across the marker/DDL transaction boundary. `chok
  migrate status --check` reports pending, dirty, checksum drift, missing,
  unverified, out-of-order, renamed and temporarily fenced migrations. Version-scoped `repair
  retry|mark-applied|accept-drift` actions require an inspected checksum and
  operational reason. Legacy three-column ledgers remain readable and adopt
  a trust-on-first-use checksum baseline on the next `up`.

  pools.** The sqlite branch now opens through
  `github.com/glebarez/sqlite` (modernc build — no CGO, so chok apps

  the CGO `gorm.io/driver/sqlite`. File databases run a read/write
  split: writes serialize on one dedicated connection
  (`_txlock=immediate`, fair Go-side queueing — SQLite only ever
  admits one writer), reads run in parallel on a WAL-snapshot pool

  **Breaking:** mattn-style DSN parameters in `db.sqlite.path`
  (`_synchronous=`, `_sync=`, `_busy_timeout=`, `_timeout=`,
  `_journal_mode=`, `_journal=`, `_foreign_keys=`, `_fk=`) are
  rejected at startup with a pointer to the `_pragma=name(value)`
  form — the pure-Go driver would otherwise silently ignore them.
  `_txlock=` keeps its spelling. Apps that relied on SQLite ignoring
  declared foreign keys must fix their data or disable enforcement
  explicitly (`?_pragma=foreign_keys(0)`).

  territory — see design.md §7.3.
- **`chok.Router` / `chok.Kernel` / `chok.Middleware` /
  `chok.Component`: application code speaks the chok vocabulary.**
  The kernel contracts that appear in root-package signatures are now
  aliased in the root package, so an app never imports `kernel` just
  to write its Routes callback:
  `chok.Routes(func(r chok.Router, k chok.Kernel) error { ... })`.
  Aliases are identical types — existing code spelling `kernel.Router`
  keeps compiling. Module authors (Descriptor, Mounter, ...) keep the
  kernel spelling; that is the extension surface, not the assembly
  surface. `examples/blog`, the README snippets and the `chok init`
  scaffold use the new spelling.
- CI/supply chain: `govulncheck` workflow (PRs + weekly sweep, tool
  version pinned), Dependabot (gomod + github-actions), all GitHub
  Actions pinned to commit SHAs, and SPDX SBOMs (syft) attached to
  release archives.
- docs: `SECURITY.md` (private vulnerability reporting, supported
  versions, disclosure process) and `CONTRIBUTING.md` (development
  loop, testing discipline, and the canonical release runbook —
  `make tag` now points at it).

### Removed

- **`web.Router` and `web.Middleware` type aliases.** They had zero
  uses in the framework, the examples or the scaffold — the canonical
  app-facing spelling is now `chok.Router` / `chok.Middleware`, and
  the contract stays `kernel.Router` / `kernel.Middleware` (which the
  aliases were identical to, so any downstream reference is a
  mechanical respell).

### Features

* **kernel:** single-actor control plane — all lifecycle transitions on one goroutine, lock-free read snapshots, topological init/close, mount → serve → drain phases, coalescing reload, typed event bus
* **assembly:** explicit typed modules (`chok.Use` / `chok.Override` / `chok.Routes`), Descriptor-declared dependencies, registration-disabled model with defined semantics, `(kind, instance)` multi-instance support
* **conf:** immutable RCU snapshots (no live pointers, no torn reads), `reload:"hot|restart"` field tags with framework-enforced dispatch, `sensitive:"true"` redaction, typed business sections via `chok.Section[T]`
* **web:** stdlib `ServeMux` HTTP stack (gin removed) — route table drives the generated OpenAPI spec, written-tracking middleware chain, trusted-proxy ClientIP, h2c and shutdown budget
* **db/store:** `*db.DB` thin handle hides gorm; context-propagated transactions (`RunInTx` + `AfterCommit`); Postgres day-one (pgx); versioned forward-only migrations with cross-process locking and the battery-table whitelist; store events anchored to commit via `store.WithBus`
* **batteries:** account (explicit OAuth provider assembly, login rate limiting), authz/casbin (bootstrap seeding in Migrate, decision audit with the audit-required truth table), audit (async sink, purge cron, fail-closed admin API), scheduler (`Server` with stop budget), cache (otter + redis + breaker), redis (TLS/CA)
* **cli:** `chok init` (v2 scaffold that boots as generated), `chok sync` (chok.yaml → assembly codegen), `chok migrate`, `chok docs gen` (components tables / config reference / JSON Schema), `chok openapi export`
* **ci gates:** `docs gen --check`, `sync --check`, `apidiff` vs release baseline (API change requires a changelog entry), examples smoke boot

### ⚠ Breaking changes

* module path `github.com/zynthara/chok` → `github.com/zynthara/chok/v2`; `component/`, `parts/`, `server/`, `config/` packages removed; gin and badger leave the dependency tree
